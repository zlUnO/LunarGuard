# LunarGuard

**LunarGuard** is a Lua 5.1 source-code protector. It transforms a plain Lua
script into a heavily obfuscated one by applying a pipeline of independent
obfuscation passes — variable renaming, string and number encryption, control-flow
flattening, dead-code injection, expression splitting, anti-debug traps and a
custom bytecode **virtual machine** that replaces your functions with encrypted
bytecode executed by a generated interpreter.

> Target runtime: **Lua 5.1** (GameSense / CS:GO cheat environment compatible).

---

## What it does

| Pass | What it changes | Status |
|------|-----------------|--------|
| Variable Renaming | Locals become unreadable hexadecimal identifiers | ✅ |
| String Encryption | Literals are replaced by runtime-decrypted tables | ✅ |
| Number Encoding | Numeric literals become arithmetic expressions | ✅ |
| Dead Code Injection | Unreachable junk blocks are sprinkled in | ✅ |
| Control Flow | Statements wrapped in opaque predicates / `do` blocks | ✅ |
| Expression Splitting | Complex expressions broken into temp locals | ✅ |
| Anti-Debug | Detects debug hooks and `debug.getinfo` probing | ✅ |
| Anti-Tamper | Runtime integrity hash guards the script | ✅ |
| Bytecode Virtualization | Functions compiled to a private VM + interpreter | ✅ |

Every pass can be toggled independently, so you can mix a light or an
aggressive protection profile.

## Building

```bash
git clone https://github.com/zlUnO/LunarGuard.git
cd LunarGuard
dotnet build -c Release
```

Requires the **.NET 9 SDK**. Runs on Windows, Linux and macOS.

To produce a standalone, self-contained Windows executable:

```bash
dotnet publish LunarGuard.CLI -c Release -r win-x64 --self-contained
```

## Using it

### Command line

```bash
# Obfuscate with every pass enabled (default)
dotnet run --project LunarGuard.CLI -- obfuscate script.lua

# Choose an output path
dotnet run --project LunarGuard.CLI -- obfuscate script.lua -o out.lua

# Turn passes off
dotnet run --project LunarGuard.CLI -- obfuscate script.lua --no-vm --no-antidebug

# Custom string-encryption key & verbosity
dotnet run --project LunarGuard.CLI -- obfuscate script.lua --key "mySecret" -v
```

| Option | Alias | Description |
|--------|-------|-------------|
| `[input]` | | Input Lua script path |
| `--output` | `-o` | Output path (default `input.obfuscated.lua`) |
| `--no-rename` | | Disable variable renaming |
| `--no-strings` | | Disable string encryption |
| `--no-numbers` | | Disable number encoding |
| `--no-deadcode` | | Disable dead-code injection |
| `--no-cflow` | | Disable control-flow obfuscation |
| `--no-split` | | Disable expression splitting |
| `--no-antidebug` | | Disable anti-debug checks |
| `--no-vm` | | Disable bytecode virtualization |
| `--key` | | String-encryption key (auto-generated by default) |
| `--deadcount` | | Number of dead-code blocks (default 5) |
| `--verbose` | `-v` | Show enabled/disabled passes |

> `obfuscate` also accepts the aliases `ob` and `protect`.

### GUI (Windows)

```bash
dotnet run --project LunarGuard.GUI
```

A WPF interface for drag-and-drop obfuscation with live pass toggles.

## How the VM works

When virtualization is on, each function body is compiled to a flat list of
opcodes and operands, XOR-encrypted, and shipped as a table. A single
shared interpreter (`loadstring`-compiled, registered once per file) decodes and
executes the bytecode. The interpreter is closure-based: handlers close over a
shared stack, register table and program counter, so every virtualized function
reuses the same engine.

Functions that capture outer-scope variables (closures / upvalues) are
**not** virtualized — the VM has a flat register model and would otherwise
resolve captured locals to `nil`. They still pass through the other obfuscation
passes.

---

## Important constraints

- **Lua 5.1 only.** `table.pack`/`table.unpack` and `load` are rewritten to
  the 5.1 equivalents (`{}` / `unpack` / `loadstring`).
- The `debug` library is referenced defensively (wrapped in `pcall`) because
  GameSense does not expose it.
- Virtualization requires self-contained functions. A function that declares a
  method on a local table (`local t = {}; function t:f() end`) keeps its
  original definition so the binding stays in scope.
